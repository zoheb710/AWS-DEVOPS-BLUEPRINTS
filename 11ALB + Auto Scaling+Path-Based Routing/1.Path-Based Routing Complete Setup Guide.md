# 🚀 AWS ALB + Auto Scaling: Complete Setup Guide
### Path-Based Routing for Mobile -  Car -  Bike Services on Ubuntu

***

## 🏗️ Architecture Blueprint

```
                        🌐 Internet
                             ↓
              ┌──────────────────────────────┐
              │   ALB — Application Load     │
              │        Balancer              │
              └──────────────────────────────┘
                 ↙            ↓            ↘
           /mobile           /car          /bike
              ↓               ↓               ↓
         📱 TG-Mobile    🚗 TG-Car       🏍️ TG-Bike
              ↓               ↓               ↓
         ASG-Mobile       ASG-Car         ASG-Bike
         (Ubuntu EC2)    (Ubuntu EC2)    (Ubuntu EC2)
```

> 💡 **Each ASG has:** Scale OUT at 20% CPU ➕ | Scale IN at 10% CPU ➖

***

## 📦 Phase 1 — Prepare Ubuntu EC2 Instances

### 🔷 Step 1: Launch 3 Ubuntu Base Instances

Launch **3 separate Ubuntu 22.04 LTS EC2 instances** (one per service) to build your AMIs for Launch Templates. 

SSH into each instance and run the setup script:

**📱 Mobile Instance:**
```bash
sudo apt update -y
sudo apt install apache2 -y
sudo mkdir -p /var/www/html/mobile
echo "<h1>Mobile Service - $(hostname -f)</h1>" | sudo tee /var/www/html/mobile/index.html
sudo systemctl enable apache2 && sudo systemctl start apache2
```

**🚗 Car Instance:**
```bash
sudo apt update -y

echo "<h1>Car Service - $(hostname -f)</h1>" | sudo tee /var/www/html/car/index.html
```

**🏍️ Bike Instance:**

## 🔐 Phase 2 — Security Groups

### 🔷 Step 2: Create 2 Security Groups

**🛡️ SG-1: `ALB-SG`** *(for the Load Balancer)*

| Direction | Protocol | Port | Source |
|---|---|---|---|
| ⬇️ Inbound | HTTP | 80 | `0.0.0.0/0` |
| ⬆️ Outbound | All | All | Anywhere |

**🛡️ SG-2: `EC2-SG`** *(for EC2 Instances)*

| Direction | Protocol | Port | Source |
|---|---|---|---|
| ⬇️ Inbound | HTTP | 80 | `ALB-SG` only |
| ⬇️ Inbound | SSH | 22 | Your IP |
| ⬆️ Outbound | All | All | Anywhere |

> ⚠️ EC2s should **only accept traffic from the ALB**, never directly from the internet.

## 🎯 Phase 4 — Target Groups

### 🔷 Step 4: Create 3 Target Groups

Go to: **EC2 → Target Groups → Create Target Group**

| Setting | 📱 TG-Mobile | 🚗 TG-Car | 🏍️ TG-Bike |
|---|---|---|---|
| **Target Type** | Instances | Instances | Instances |
| **Protocol** | HTTP | HTTP | HTTP |
| **Port** | 80 | 80 | 80 |
| **Health Check Path** | `/mobile/index.html` | `/car/index.html` | `/bike/index.html` |
| **Healthy Threshold** | 2 | 2 | 2 |
| **Unhealthy Threshold** | 3 | 3 | 3 |
| **Interval** | 30 sec | 30 sec | 30 sec |

> ✅ **Do NOT register instances manually** — your Auto Scaling Groups will auto-register them to Target Groups. 
***

## ⚖️ Phase 5 — Application Load Balancer

### 🔷 Step 5: Create the ALB

Go to: **EC2 → Load Balancers → Create Load Balancer → Application Load Balancer**

| Field | Value |
|---|---|
| **Name** | `ALB-Services` |
| **Scheme** | Internet-facing |
| **IP Type** | IPv4 |
| **VPC** | Your VPC |
| **Subnets** | ✅ Min 2 Public Subnets (different AZs) |
| **Security Group** | `ALB-SG` |
| **Listener** | HTTP : 80 |
| **Default Action** | Forward to `TG-Mobile` |

Click **Create Load Balancer** ✅ 
***

### 🔷 Step 6: Configure Path-Based Routing Rules

Go to: **ALB → Listeners → HTTP:80 → Manage Rules → Add Rule** 

| Priority | 🛣️ Path Condition | ➡️ Action |
|---|---|---|
| **1** | `/mobile*` | Forward → `TG-Mobile` |
| **2** | `/car*` | Forward → `TG-Car` |
| **3** | `/bike*` | Forward → `TG-Bike` |
| **Default** | Everything else | Forward → `TG-Mobile` |

> 💡 Priority order matters — ALB evaluates rules top to bottom and uses the **first match**.
***

## ⚡ Phase 7 — Scaling Policies

Do this **for each ASG** (ASG-Mobile, ASG-Car, ASG-Bike) individually.

***

### 🔷 Step 8: 🔼 Scale OUT Policy — Add Instance at 20% CPU

Go to: **ASG → Automatic Scaling → Create Dynamic Scaling Policy**

| Setting | Value |
|---|---|
| **Policy Type** | `Target Tracking Scaling` |
| **Policy Name** | `ScaleOut-20CPU` |
| **Metric Type** | Average CPU Utilization |
| **Target Value** | `20` % |
| **Instance Warmup** | 60 seconds |
| **Disable Scale-In** | ✅ YES (scale-in handled separately) |

> 📌 Target Tracking continuously monitors CPU and **automatically adds a new instance** when average CPU ≥ 20% across the group.

***

### 🔷 Step 9: 🔽 Scale IN Policy — Remove Instance at 10% CPU

This requires **two sub-steps**: a CloudWatch Alarm + a Step Scaling Policy.

#### ☁️ Sub-Step A: Create CloudWatch Alarm

Go to: **CloudWatch → Alarms → Create Alarm**

| Setting | 📱 Mobile | 🚗 Car | 🏍️ Bike |
|---|---|---|---|
| **Metric** | `ASG-Mobile` → CPUUtilization | `ASG-Car` → CPUUtilization | `ASG-Bike` → CPUUtilization |
| **Statistic** | Average | Average | Average |
| **Period** | 5 minutes | 5 minutes | 5 minutes |
| **Condition** | `CPU < 10%` | `CPU < 10%` | `CPU < 10%` |
| **Alarm Name** | `LowCPU-Mobile-10` | `LowCPU-Car-10` | `LowCPU-Bike-10` |

#### 📉 Sub-Step B: Create Step Scaling Policy

Go to: **ASG → Automatic Scaling → Create Dynamic Scaling Policy**

| Setting | Value |
|---|---|
| **Policy Type** | `Step Scaling` |
| **Policy Name** | `ScaleIn-10CPU` |
| **CloudWatch Alarm** | `LowCPU-[Service]-10` |
| **Step Action** | Remove `1` instance when `0 ≤ CPU < 10%` |
| **Cooldown** | 120 seconds |

> 🔄 Step Scaling gives you fine-grained control — you define exactly **how many instances to remove** per threshold band. 
***

## 📊 Scaling Policy at a Glance

| 🏷️ Policy Name | 📐 Type | ⚡ Trigger | 🎬 Action | ⏱️ Cooldown |
|---|---|---|---|---|
| `ScaleOut-20CPU` | Target Tracking | CPU ≥ 20% | ➕ Add 1 instance | 60 sec warmup |
| `ScaleIn-10CPU` | Step Scaling | CPU < 10% | ➖ Remove 1 instance | 120 sec |

```bash
curl http://<ALB-DNS>/car/index.html      # ✅ Should return Car EC2
```

SSH into any EC2 instance and run:

```bash
sudo apt install stress -y

```

> 👀 Watch **ASG → Activity** tab — a new instance should auto-launch within ~2 minutes when CPU hits 20%.
